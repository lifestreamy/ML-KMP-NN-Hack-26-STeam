# USE CASES & USER FLOWS
> Version: 1.4

## Пользователь
Геолог / лаборант НИИ. 

---

## Архитектурное ограничение: Active Learning
Сама тренировка (дообучение YOLO/SAM) на лету НЕ запускается. Система только собирает датасет: исправления геологов кладутся в Volume, а дообучение предполагается внешним MLOps пайплайном по расписанию.

---

## UC-1: Очередь и Пакетная обработка (Основной флоу)

**Цель:** Загрузить пачку шлифов, контролируемо встать в очередь, получить результат асинхронно.

**User Flow:**
1. Геолог выбирает пачку файлов (TIFF/PNG/JPEG).
2. Нажимает "Анализировать".
3. UI проверяет статус сервера (GET `/queue/status`).
4. Если место есть, UI отправляет файлы. Ktor валидирует лимиты, принимает файлы, отвечает `200 OK` с массивом UUID задач.
5. В UI создаются карточки задач со статусом `В очереди (N перед вами)`.
6. Задачи обрабатываются по одной (строго FIFO). 
7. Статусы карточек обновляются через SSE: `Обработка` → `Готово` (или `Ошибка`).
8. Готовая карточка кликабельна: открывает Canvas с масками и метриками.

**Управление задачами (Отмена):**
- Геолог может выбрать чекбоксами несколько карточек в очереди.
- Нажимает "Отменить выбранные".
- UI шлет `DELETE /tasks` с массивом IDs.
- Ktor удаляет их из очереди. Если задача уже обрабатывается в Python, мы её не прерываем (чтобы избежать гонок состояний "отменили не то"), Ktor просто выкинет результат, не отдавая клиенту.

## UC-2: Экспертная разметка (Сбор датасета)

**Цель:** Скорректировать ошибки нейросети и сохранить в БД для будущих дообучений.

**User Flow:**
1. На экране результатов геолог видит неверную разметку.
2. Включает режим "Корректировка".
3. Выделяет полигоном область, указывает правильный класс.
4. Нажимает "Сохранить правки".
5. Ktor Gateway сохраняет в Volume `/dataset/feedback` **пару файлов** с одинаковым UUID в названии: картинку (`<task_id>.tiff`) и метаданные (`<task_id>.json`). Внутри JSON также явно прописано имя файла-картинки.
